HTML Entities Explained

HTML entities are the mechanism that lets you show characters like <, >, and & as literal text instead of having the browser treat them as markup. Get this wrong and your page renders half-broken — or worse, opens a door to cross-site scripting. This guide explains what entities are, which characters you must escape, the difference between named and numeric forms, and when to reach for escaping at all.

Why HTML needs escaping

An HTML parser reads a document looking for structural signals. A < starts a tag. An & starts an entity or character reference. If your content happens to contain one of those characters, the parser can't tell your data apart from the page's structure.

Say you want the words a < b to appear on the page. Written raw, the browser sees < b and starts hunting for a tag named b. The fix is to escape the character: write &lt; and the browser renders a literal less-than sign.

Raw     : if (a < b) doThing();
Escaped : if (a &lt; b) doThing();

The characters you must escape

Only a small set is truly mandatory. The rest is context.

CharacterEntityEscape when…
&&amp;Always — it begins every entity
<&lt;Always — it begins every tag
>&gt;In text (safe practice; prevents edge cases)
"&quot;Inside a double-quoted attribute value
'&#39;Inside a single-quoted attribute value

A reliable rule: escape &, <, and > in any text you insert, and also escape quotes whenever you build an attribute value from untrusted input.

Named vs numeric entities

Every entity comes in two flavors, and they render identically:

  • Named — a human-friendly label, e.g. &copy; for ©, &mdash; for —, &nbsp; for a non-breaking space. Easy to read, but only defined for a fixed set of characters.
  • Numeric — the character's Unicode code point in decimal (&#169;) or hexadecimal (&#xA9;). Works for any character, including emoji.
💡 When in doubt, numeric entities always work. Named entities are nicer to read but you have to know the name exists. For the five markup-critical characters, prefer the well-known names: &amp;, &lt;, &gt;, &quot;.

Escaping and security (XSS)

The most important reason to escape is safety. When you drop user-supplied text straight into a page, an attacker can inject <script> tags that run in your visitors' browsers — a cross-site scripting (XSS) attack. Escaping < and & turns that payload into inert text: &lt;script&gt; displays as harmless characters instead of executing. Always escape untrusted input at the point you insert it into HTML.

Do you need to escape accented letters and emoji?

Usually no. If your page declares <meta charset="utf-8"> — which it should — you can write café, €, and 😀 directly. Escaping every non-ASCII character is optional and mainly helps legacy systems or email templates that can't be trusted to preserve UTF-8. Frequently asked questions

Which characters must be escaped? Always & and <; also > in text, and quotes inside attribute values.

Named or numeric? Both render the same. Numeric works for every character; named is more readable but limited to a defined set.

Is escaping the same as encryption? No — it's reversible formatting with zero secrecy. Its job is correct rendering and XSS prevention.

Do I need to escape emoji and accents? Not on a UTF-8 page. Escape only the markup-critical characters unless a legacy pipeline forces otherwise.
