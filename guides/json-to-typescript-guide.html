How to Convert JSON to TypeScript Interfaces

Most TypeScript projects consume JSON they didn't define — a REST endpoint, a third-party webhook, a config file. Parsing that JSON with JSON.parse gives you a value typed as any, which throws away everything TypeScript is good at. The fix is to write an interface that describes the shape, so the compiler can catch typos and missing fields. Writing those interfaces by hand is tedious; generating them from a sample is instant. This guide shows how the conversion works and how to make the result trustworthy.

The basic idea

A JSON to TypeScript converter reads one representative value and maps each JSON type to its TypeScript equivalent: a JSON string becomes string, a number becomes number, a boolean becomes boolean, an object becomes a named interface, and an array becomes a typed array. You paste JSON, you get a ready-to-use type.

// Input JSON
{ "id": 1, "name": "Ada", "active": true }

// Generated TypeScript
export interface Root {
  id: number;
  name: string;
  active: boolean;
}

Nested objects become their own interfaces

When a property holds another object, it is cleaner to give that object its own named interface and reference it, rather than inlining a deeply nested shape. A good generator does this automatically, naming the child interface after its key.

export interface Address {
  city: string;
  zip: string;
}

export interface User {
  id: number;
  address: Address;
}

Arrays and unions

Arrays are the tricky part, because JSON arrays can hold anything. The converter inspects the elements and infers a single element type:

  • An array of strings becomes string[].
  • An array of mixed primitives becomes a union: (string | number)[].
  • An array of objects is merged into one interface. If a key appears in some elements but not all, it becomes optional.
  • An empty array can't be inferred, so it falls back to unknown[] — a signal to look closer.
💡 Feed the generator several representative rows, not just one. If your API sometimes omits a field or returns null, a single sample won't show it — a handful of rows will, and the merged type captures the optional keys and unions correctly.

Handling null: optional vs explicit

JSON's null maps to two reasonable TypeScript shapes, and which you want depends on your API:

ChoiceGenerated fieldUse when…
Optionalscore?: numberThe field may be absent or null and callers should guard for both
Explicit nullscore: number | nullThe field is always present but can carry a null value

The generator on this site exposes a null → optional toggle so you can produce whichever your endpoint actually returns.

interface vs type

You'll see both in the wild. For describing a plain JSON object shape they are interchangeable: interface User { … } and type User = { … } compile to the same structural check. Interfaces support declaration merging and are the conventional choice for object shapes; type aliases shine for unions and intersections. Pick whatever matches your codebase — the generator can emit either.

Review before you commit

A generated type is a strong starting point, not gospel. It reflects only the sample you pasted. Before you rely on it, sanity-check three things: are any "always present" fields actually optional in production, are numeric-looking strings really strings (IDs often are), and do enum-like fields deserve a literal union such as "active" | "archived" instead of a bare string. A minute of review turns a decent type into an accurate one.

Frequently asked questions

How do I convert JSON to a TypeScript interface? Paste a representative object or API response into the generator; it maps each value to its TypeScript type, extracts nested objects into named interfaces, and infers array element types.

Should I use interface or type? Either works for a data shape. Interfaces are the common convention for objects; type aliases are handier for unions and intersections.

How are null values handled? You choose: make the property optional (name?: string) or keep an explicit null in the type (name: string | null).

Is one sample enough? Often not. Paste several representative rows so optional keys and unions are detected, then review the result.
