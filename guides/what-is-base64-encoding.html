What Is Base64 Encoding?

Base64 is a way of representing binary data using only 64 printable ASCII characters. It shows up everywhere in software — in data URLs, email attachments, API tokens, and configuration files — yet it's frequently misunderstood. This guide explains what Base64 actually does, why it is not a security measure, and the situations where reaching for it is the right call.

How Base64 works

Computers store everything as bytes, but many systems — email, URLs, JSON, XML — were designed to carry text, not arbitrary binary. Base64 bridges that gap. It takes raw bytes and rewrites them using a fixed alphabet of 64 characters: A–Z, a–z, 0–9, plus + and / (with = used as padding).

Mechanically, Base64 reads the input three bytes (24 bits) at a time and splits those 24 bits into four 6-bit groups. Each 6-bit group (a value from 0 to 63) maps to one character in the alphabet. So every 3 bytes of input become 4 characters of output, which is why Base64 data is roughly 33% larger than the original.

For example, the text Hi encodes to SGk=. The trailing = is padding that signals the original input didn't fill a complete 3-byte group.

Base64 is not encryption

⚠️ Base64 provides zero confidentiality. Anyone can decode it instantly — no key required. It's an encoding (a reversible format change), not encryption (which protects data with a secret). Never use Base64 to "hide" passwords, API keys, or any sensitive value.

It's an easy trap because Base64 output looks scrambled. But scrambled-looking is not the same as secret. If you need to protect data, use real encryption (such as AES) or transport security (TLS/HTTPS) instead.

When to use Base64

  • Embedding small files inline — data URLs like data:image/png;base64,... let you include a small icon or font directly in HTML or CSS without a separate request.
  • Putting binary in text-only formats — storing an image or file blob inside JSON, XML, or a YAML config that can only hold text.
  • Email attachments — the MIME standard uses Base64 to send binary files over text-based email protocols.
  • Encoding token segments — the header and payload of a JWT are Base64URL-encoded (a URL-safe variant that swaps +/ for -_).

When NOT to use Base64

  • For large files — the 33% size overhead and lost browser caching usually outweigh the convenience.
  • For security — see above; it protects nothing.
  • When a binary-safe channel already exists — if you can send raw bytes, encoding them is wasted effort.
