How to Format SQL Queries (and Why Readable SQL Matters)

SQL has a habit of growing into dense, one-line monsters — a SELECT with six joins, a stack of conditions, and a GROUP BY all crammed onto a single line that scrolls off the screen. It runs fine, but nobody can read it. Formatting your SQL — giving each clause its own line, indenting consistently, and casing keywords predictably — turns that wall of text into something a teammate (or future you) can scan in seconds. This guide covers what "formatted SQL" means, why it matters, and how to do it in one click.

Why readable SQL matters

Queries are read far more often than they are written. A well-formatted query pays off every time someone opens it:

  • Faster review and debugging. When clauses line up, you can spot a missing join condition or a stray AND at a glance instead of parsing a paragraph.
  • Cleaner diffs. If each column and condition sits on its own line, a code review shows exactly which line changed — not one giant modified line where you have to hunt for the edit.
  • Fewer bugs. Consistent layout makes logic errors visible: an accidentally commented-out filter or a condition attached to the wrong clause stands out.
  • Shared style. A team that formats SQL the same way spends zero energy arguing about whitespace and reads each other's queries effortlessly.

Before and after

Here's a typical query as it often arrives — everything on one line:

select u.id, u.name, count(o.id) as orders from users u left join orders o on o.user_id=u.id where u.active=true and o.created_at > '2024-01-01' group by u.id, u.name having count(o.id) > 3 order by orders desc limit 10;

And the same query after formatting — clauses broken out, columns indented, keywords uppercased:

SELECT
  u.id,
  u.name,
  count(o.id) AS orders
FROM
  users u
  LEFT JOIN orders o ON o.user_id = u.id
WHERE
  u.active = true
  AND o.created_at > '2024-01-01'
GROUP BY
  u.id,
  u.name
HAVING
  count(o.id) > 3
ORDER BY
  orders DESC
LIMIT
  10;

Same result set, same performance — but now the join, the two filters, and the HAVING threshold are each on their own line, so the shape of the query is obvious.

The three choices that define SQL style

Almost every SQL style decision comes down to three knobs, all of which the formatter exposes:

1. Indentation

Two spaces is the common default; some teams prefer four, and some use tabs. Consistency matters more than the exact width — pick one and apply it everywhere. Indentation is what visually nests columns under SELECT and conditions under WHERE.

2. Keyword case

The long-standing convention is UPPERCASE keywords (SELECT, FROM, WHERE) against lowercase identifiers, because it separates the language from your table and column names at a glance. Lowercase-everything is also popular in modern codebases. The formatter lets you choose UPPER, lower, or preserve — again, the win is picking one and being consistent.

3. Clause layout

Each major clause (SELECT, FROM, WHERE, GROUP BY, ORDER BY) starts a new line, and the items under it — columns, joins, conditions — are indented one level. This "one thing per line" layout is what makes diffs clean and errors visible.

Dialect notes

SQL is a family of related dialects, not one language. Keywords and syntax differ between databases, so telling the formatter which dialect you're using produces better results:

DialectNotes
PostgreSQLRich type system, RETURNING, :: casts, and ILIKE.
MySQL / MariaDBBacktick-quoted identifiers and LIMIT offset syntax.
T-SQL (SQL Server)Uses TOP instead of LIMIT and square-bracket identifiers.
BigQueryStandard SQL with arrays, structs, and backtick-quoted table paths.
SQLiteLightweight subset; flexible typing.

If you're not sure, Standard SQL is a safe default that handles the common ANSI syntax well.

Formatting is purely cosmetic — it never changes what your query returns or how the database plans it. It only changes whitespace, line breaks, and keyword case, so it's completely safe to run on any query.

Frequently asked questions

Does formatting SQL change how the query runs?

No. SQL ignores extra whitespace and newlines, and keyword case is not significant. Formatting only affects how the query looks to humans — the parser, planner, and results are identical.

Should SQL keywords be uppercase or lowercase?

Either works; consistency is what counts. Uppercase keywords are the traditional choice because they stand out from lowercase table and column names, but many modern teams write everything lowercase. Pick one convention and apply it across your codebase.
