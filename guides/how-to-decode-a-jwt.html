How to Decode a JWT (JSON Web Token)

A JSON Web Token, or JWT (often pronounced "jot"), is a compact, URL-safe string used to carry claims between two parties — most commonly between an authentication server and a client app. If you have ever inspected a request and seen a long string of seemingly random characters with two dots in it, that's almost certainly a JWT. This guide explains how a JWT is structured, how to decode one to see what's inside, and the security rules you should never break.

What a JWT looks like

A JWT is three Base64URL-encoded segments joined by dots:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkphbmUgRG9lIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

The three parts are the header, the payload, and the signature, written as header.payload.signature.

What's inside a JWT

  • Header — JSON describing how the token is signed, e.g. {"alg":"HS256","typ":"JWT"}. The alg field names the signing algorithm.
  • Payload — JSON containing the claims (the actual data). Standard claims include sub (subject), iat (issued-at), exp (expiry), and iss (issuer), alongside any custom fields your app adds.
  • Signature — a cryptographic signature over the header and payload. It lets a server confirm the token hasn't been altered and was issued by someone holding the signing key.

How to decode a JWT, step by step

  1. Split the token on the dot characters into its three parts.
  2. Take the first part (the header) and Base64URL-decode it. You'll get readable JSON.
  3. Do the same with the second part (the payload).
  4. Read the claims. If you see exp, iat, or nbf, those are Unix timestamps — convert them to human-readable dates to check validity windows.

The header and payload are only encoded, not encrypted, so anyone can read them. The signature is the only part that requires a key, and it cannot be reversed into readable text — it's there to be verified, not decoded.

Security note: never paste secrets

🔒 Decoding a JWT reveals everything in its payload. Because the payload is readable by anyone, you should never store sensitive data (passwords, API keys, personal secrets) inside a token. And when troubleshooting, never paste a live production token into a third-party online tool you don't trust — a valid, unexpired token can be replayed to impersonate the user.

Our JWT Decoder runs entirely in your browser: the token never leaves your device and is never sent to a server. Even so, prefer using expired or test tokens when you can, and rotate any token you suspect has been exposed.

Decoding vs. verifying

Decoding tells you what a token says. Verifying tells you whether you can trust it. Verification requires recomputing the signature with the signing secret (for HS256) or public key (for RS256) on a trusted server. A decoder alone cannot tell you whether a token is authentic — only what claims it carries and whether, on paper, it has expired.
